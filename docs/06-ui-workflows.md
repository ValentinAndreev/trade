# UI Workflows

## Обзор

Этот раздел описывает, как пользователь работает с системой в интерфейсе.

## Сценарий 1. Открыть рабочее пространство

При заходе пользователь видит две верхнеуровневые страницы:

- `Main`
- `Graph`

`Main` подходит для быстрого мониторинга. `Graph` это основной аналитический режим.

## Сценарий 2. Работать с главной страницей

На `Main` есть четыре секции:

- `Crypto`
- `World Indices`
- `Forex`
- `Commodities`

Что можно делать:

- добавлять и удалять инструменты из каждой секции;
- открыть график кликом по crypto tile;
- видеть базовые изменения цены, high/low и время обновления.

Важно:

- crypto tiles живут через `dashboard symbols`;
- индексы/forex/commodities живут через `market symbols`;
- это разные потоки данных и разные серверные конфигурации.

## Сценарий 3. Открыть график по инструменту

Обычно путь такой:

1. На `Main` выбрать crypto tile.
2. UI отправляет событие открытия chart tab.
3. В `Graph` создается или активируется tab.
4. На panel появляется price overlay для выбранного symbol.
5. История загружается через `/api/candles`.
6. Новые свечи приходят через ActionCable.

## Сценарий 4. Настроить chart tab

В графическом tab пользователь может:

- переименовать tab (кликом на названии);
- добавить или удалить panel;
- изменить timeframe panel;
- добавить overlay;
- переключить overlay между `price`, `volume`, `indicator`;
- поменять chart type;
- управлять цветом и прозрачностью;
- скрыть overlay без удаления.

Ключевая модель:

- `Tab` содержит `Panel[]`;
- каждая `Panel` содержит `Overlay[]`;
- активный overlay редактируется в боковой панели.

Важно про main chart layer:

- у chart tab есть `primary panel`;
- у каждой panel первый overlay считается `primary overlay`;
- `primary overlay` это основной инструмент панели;
- его нельзя удалить как обычный overlay;
- он исчезает только если удалить саму panel или весь tab;
- его нельзя перевести из `price` в другой режим.

Все дополнительные overlays уже являются вторичными слоями и могут свободно добавляться, скрываться и удаляться.

## Сценарий 5. Добавить индикатор

Пользователь выбирает overlay режима `indicator` и настраивает:

- тип индикатора;
- параметры;
- источник данных;
- overlay, к которому индикатор привязан.

Список индикаторов фильтруется по источнику:

| Фильтр | Что показывает |
| --- | --- |
| All | Все доступные источники |
| Client | Клиентские индикаторы (JS-вычисления) |
| Server | Серверные индикаторы (через `/api/indicators`) |
| Data | Макроэкономические данные (Data Sources) |

**Источники:**

- **Client-side** — вычисляются в браузере, быстро реагируют, работают при отключенном сервере;
- **Server-side** — единый серверный API, используются в research/backtest;
- **Data Sources (macro)** — макроэкономические ряды (DXY, VIX, Fear & Greed, Fed Rate, M2, CPI).

Особенности macro overlay:

- нет параметров `period` или `source` — в боковой панели они не отображаются;
- данные загружаются из `/api/macro_series` один раз при добавлении оверлея;
- при выборе в списке получает собственную шкалу (не смешивается со шкалой price).

## Сценарий 6. Построить standalone data tab

Data tab используется для табличного анализа исторических данных.

Обычно пользователь:

1. Создает data tab.
2. Выбирает символ и timeframe.
3. Добавляет нужные колонки.
4. При необходимости задает диапазон времени.
5. Получает таблицу с OHLCV и вычисляемыми полями.

Колонки могут быть такими:

- базовые OHLCV;
- indicator columns (клиентские или серверные);
- change columns (процентное/абсолютное изменение);
- formula columns (пользовательское выражение);
- instrument columns по другому symbol;
- **macro columns** — значение макро-индикатора (DXY, VIX, Fear & Greed, Fed Rate, M2, CPI) на каждом баре, с LOCF-интерполяцией.

Это самостоятельный режим анализа: он не обязан быть связан с chart tab.

## Сценарий 7. Создать linked data tab из chart tab

Связанный data tab создается от chart tab и наследует его контекст.

Что происходит при создании:

1. За основу берется `primary panel` chart tab.
2. Базовый symbol берется из `primary overlay` этой panel.
3. Базовый timeframe тоже берется из этой panel.
4. В data tab добавляются стандартные OHLCV-колонки.
5. Индикаторы, уже присутствующие на chart tab, добавляются как indicator-columns.
6. Data tab ставится рядом с chart tab в общей визуальной группе.

Что дает linked mode:

- data tab следует за базовым symbol/timeframe связанного графика;
- изменение timeframe синхронизируется между chart panel и data tab;
- click по строке таблицы перемещает связанный график к тому же времени;
- conditions и system markers можно отрисовать обратно на linked chart;
- для cache-only сценариев данные могут обновляться из локального candle cache без полного server reload.

Практически это основной режим "сначала собрал график, потом построил по нему таблицу".

## Сценарий 8. Настроить conditions

На data tab пользователь создает правила вроде:

- price выше уровня;
- одно поле пересекло другое;
- индикатор вышел из диапазона;
- формульное выражение вернуло `true`.

Condition может влиять на интерфейс несколькими способами:

- подсветить строку;
- отфильтровать выборку;
- добавить marker на график;
- закрасить цветовую зону на графике.

Если data tab связан с chart tab, conditions могут визуально переноситься на график.

## Сценарий 9. Проверить торговую идею через TradingSystem

Для этого на data tab создается `TradingSystem`:

- long entry rule;
- long exit rule;
- short entry rule;
- short exit rule;
- slippage;
- настройки отображения.

Дальше система:

- генерирует сделки;
- считает метрики;
- строит system stats tab;
- может отрисовать сигналы на графике.

Важно, как именно это выглядит в UI:

- каждая включенная система добавляет в data tab отдельную колонку;
- в колонке показываются состояния `ENTRY`, `EXIT` и `OPEN`;
- для `OPEN` отображается состояние позиции и плавающий PnL на промежуточных барах;
- при включенном `showOnChart` входы и выходы отображаются на linked chart;
- для системы можно открыть отдельный `system_stats` tab.

`system_stats` tab:

- создается рядом с data tab;
- считает статистику по сделкам на основе rows текущего data tab;
- показывает equity curve, summary metrics и список сделок.

Если data tab входит в связанную группу с chart tab, его stats tab тоже визуально идет рядом с этой группой.

## Сценарий 10. Сохранить рабочее пространство

Presets нужны для сохранения составного состояния пользователя.

При сохранении в preset попадают:

- tabs;
- active tab;
- активная страница;
- dashboard symbols;
- markets symbols.

Что важно:

- часть состояния изначально живет в `localStorage`;
- часть server-side state отдельно снимается через `/api/presets/state`;
- при применении пресета обе части собираются обратно;
- сам `preset payload` при этом лежит в БД, а не в browser storage.

## Сценарий 11. Работать с LLM-ассистентом

Ассистент доступен через отдельную страницу в интерфейсе.

Основной сценарий:

1. Открыть страницу ассистента.
2. Выбрать провайдера и ввести API-ключ через иконку настроек (один раз, хранится в БД).
3. Создать новый чат или открыть существующий.
4. Задать вопрос или запросить помощь с торговой системой.

Режимы работы при привязке к system editor:

- **system_patch** — ассистент предлагает патч к существующему YAML системы (изменения условий, параметров, модулей);
- **system_design** — ассистент генерирует новую систему с нуля.

Чтобы привязать ассистента к редактору:

1. Открыть редактор торговой системы (Research section).
2. Открыть панель ассистента рядом с редактором.
3. Ассистент видит текущее содержимое YAML и может на него ссылаться.

История чатов сохраняется в PostgreSQL. API-ключи провайдеров не хранятся в конфиге — только в БД в зашифрованном виде.

Поддерживаемые провайдеры: OpenAI, Anthropic, Gemini, OpenRouter, Mistral, xAI, Ollama, llama.cpp.

## Сценарий 12. Запустить backtest или оптимизацию

Research — серверный режим тестирования торговых систем.

Сценарий backtest:

1. Создать или открыть систему в редакторе YAML.
2. Нажать `Run` — откроется панель запуска.
3. Выбрать symbol, timeframe, диапазон дат, комиссию и проскальзывание.
4. Запустить backtest. Прогресс приходит через ActionCable.
5. Получить список сделок, метрики и equity curve.

Сценарий оптимизации:

1. Выбрать режим `Optimize`.
2. Задать диапазон значений для каждого параметра оптимизации.
3. Запустить — система перебирает все комбинации, кэшируя module results.
4. Результаты сортируются по выбранной метрике.

Использование макро-полей в условиях:

- в YAML условия могут ссылаться на `vix`, `dxy`, `fear_greed`, `fed_rate`, `m2`, `cpi`;
- макро-данные загружаются один раз в начале run и переиспользуются во всех итерациях оптимизации;
- если макро-данные за нужный период не загружены, условие с таким полем всегда вернет `false`.

## Что сохраняется автоматически

Без явного сохранения пресета локально сохраняются:

- tabs;
- active tab;
- текущая страница навигации.

Это позволяет вернуться к последнему workspace после перезагрузки браузера.

Параллельно браузер отдельно держит в `IndexedDB`:

- кэш свечей;
- кэш indicator series.

Это уже не workspace-state, а data cache для ускорения и degraded/offline mode.
