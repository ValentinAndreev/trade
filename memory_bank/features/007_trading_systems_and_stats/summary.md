# Trading Systems and Stats — Summary

> Backfilled summary of existing shipped behavior.
> Sources: `docs/03-domain-model.md`, `docs/06-ui-workflows.md`, trading system code/tests.
> Purpose: document current contract, main paths, checks and known gaps; not future work.

## Goal

Allow rule-based trading systems inside data tabs to generate trades, markers, metrics and system stats views.

## Current Contract

1. Trading system config must serialize/restore with data tab state.
2. Evaluation must be deterministic for the same rows/config.
3. Long and short flows must be represented explicitly.
4. Slippage/display settings must be applied consistently.
5. System stats tabs must handle no-trade and invalid-config states.
6. Chart markers generated by systems must remain derived visual state.

## Non-Scope

- Broker execution.
- Replacing server-side research backtests.
- Guaranteeing exchange-grade fill modeling.

## Verified By

Current verification sources are listed in `Tests`; items below describe behavior covered or constrained by the existing implementation.

- A saved system restores and can be re-evaluated.
- No-trade systems show explicit empty stats.
- Metrics/equity/trades are stable for fixed input rows.
- Changed evaluation logic has deterministic unit tests.

## Invariants Enforced By Code

- Rule evaluation and trading engine behavior are covered by dedicated frontend tests.
- System state is serialized through tab persistence.
- Trading systems remain frontend exploratory state; no backend execution path exists in this package.

## Known Gaps / Tech Debt

- System stats rendering components have limited direct tests; core calculation engine has stronger coverage than UI layout.
- Trading systems are frontend exploratory state, not backend execution; server-side parity is intentionally absent.
- Fill/slippage/market realism assumptions are simplified and should not be treated as production backtesting semantics.
