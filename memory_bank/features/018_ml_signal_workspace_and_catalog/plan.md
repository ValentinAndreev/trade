# ML Signal Workspace and Catalogue — Plan

**Spec:** `memory_bank/features/018_ml_signal_workspace_and_catalog/spec.md`

## Approach

Layer UI/data-grid/catalog expansion on top of feature 017. Do not alter 017 storage or training contracts unless a separate 017 spec change is approved. Before starting 018 implementation, complete the deferred verification checks recorded in `memory_bank/features/017_ml_signal_modules/reviews/impl.md` or explicitly record any accepted residual risk.

New Ruby public classes get RBS files in the same implementation slice that introduces them. Final verification runs `bundle exec steep check`.

Runtime code and executable tests must describe product contracts, not feature-history labels. Do not put `017`, `018` or similar task numbers in application error messages, UI text, runtime comments or spec example names; keep those references in memory-bank/docs/review artifacts.

Data-grid ML prediction rows use stable field keys derived from the column id, not from editable labels: `columnFieldKey(col)` returns `ml_prediction:<column_id>` for `ml_prediction` columns. Request/response payloads carry the same `column_id`, so duplicate model/output columns and column renames do not collide.

The data-grid prediction endpoint accepts optional `exchange` and defaults to the shared candle-query default exchange, currently `bitfinex`. Do not hard-code the literal in the controller; introduce or reuse a shared default such as `Candle::FindQuery::DEFAULT_EXCHANGE`. The backend passes the resolved exchange into 017 `Ml::InferenceService` and echoes it in the response contract.

The grid cap mirrors the final 017 `Ml::InferenceService`/`Ml::PredictionRepository` behavior: `requested_prediction_rows = candle_count * distinct(modelKey)` for one `(exchange, symbol, timeframe)` tuple. `modelOutput` and duplicate frontend columns do not multiply backend inference rows because one stored prediction tuple contains `probability`, `direction` and `confidence`; they only affect response projection and rendering. Endpoint diagnostics must not reuse the 017 single-call helper that reports `distinct_requested_models: 1`; the grid endpoint reports the actual deduplicated model count and requested outputs per model.

Model-key autocomplete uses `GET /api/ml/models/autocomplete?q=<prefix>&limit=50`, returning `{ models, meta }` with `truncated`/`has_more`, `limit` and `max_prediction_rows`. Plain `GET /api/ml/models` keeps the final 017 array response for workspace model lists unless API docs and request specs are intentionally changed in the same slice.

Expanded transform modules use a shared input-reference schema instead of ad hoc params. Single-input modules use `input`; pair modules use `left`/`right`. Each reference is one of `{ kind: "ohlcv", field: "close" }`, `{ kind: "module", alias: "module_alias", output: "value" }`, or `{ kind: "external_series", key: "vix", output: "value" }`. In 018, `ohlcv` and `module` refs are scoped to the current `(exchange, symbol, timeframe)` series; refs that try to specify another exchange, symbol or timeframe are rejected rather than silently coerced. `external_series` refs reuse the existing no-lookahead alignment: the stored `MacroSeries.ts`/series timestamp is treated as the availability timestamp, values are carried forward only after that timestamp, there is no future fill or interpolation, and values are `nil` before the first known value. Validation rejects unknown refs, missing output fields and refs whose metadata cannot prove no-lookahead behavior. This schema applies to 018 transform-like modules; the minimal 017 modules keep their existing simple params unless a future refactor retrofits them. Required cleanup: rename the `module` input-ref key `alias` to a clearer contract name such as `module_name` or `module_ref`, and update validator, runtime resolver, ML feature-window lookback logic, tests and docs together; `alias` is too overloaded for the long-term DSL contract.

Boundary normalization is deliberately narrow. Do not add `fetch_hash_value`, `read_key`, symbol/string alias fallback, blanket `to_h`/`to_s` coercion, `respond_to?`/`is_a?` defensive branches, `finite?` guards or NaN/Infinity checks for internal ML/research payloads. Controllers, DOM form handling, YAML validation and localStorage/API hydration are the only places that may normalize external shapes; numeric boundary code may reject non-finite form/API values before emitting the canonical payload. Downstream code reads the canonical shape directly and fails fast on malformed data.
