# Feature Coverage Matrix

This audit maps retrospective feature packages to PRD areas and concrete owning paths. It does not create retrospective `brief.md`, `spec.md` or `plan.md`; shipped behavior remains documented through `summary.md`.

## Audit Rules

- `summary.md` is the contract source for shipped behavior.
- `Main Implementation`, `Tests`, `Invariants Enforced By Code`, `Known Gaps / Tech Debt`, and `Verification On Change` must be present in every retrospective package.
- Future large changes to a shipped area start with forward `brief.md`, using the relevant `summary.md` and this matrix as context.
- When a path appears in multiple rows, read the most specific feature first, then the cross-cutting package.

## Overlap Rules

- Candle/ticker paths shared by dashboard and chart changes: read `001` first for data identity/sync invariants, then `002` or `003` for UI/API behavior.
- Tabs and persisted workspace state shared by chart, grid, systems, presets and workspace refactor: read the feature-specific summary first, then `015`; read `008` when `Preset.payload` compatibility is touched.
- Research YAML, system editor and LLM tools overlap on DSL validation: read `009` before `011` or `014` when condition syntax, YAML schema or validation behavior changes.
- Macro data is cross-cutting: read `012` before chart/grid/research changes that consume macro or on-chain series.

## Audit Notes

- All 16 retrospective packages have the required summary sections and focused verification commands.
- Several `> Sources:` headers intentionally include category labels such as "frontend code"; the concrete operational ownership is the `Main Implementation` list in each summary and the matrix above.
- Known weak spots are already captured in package `Known Gaps / Tech Debt`; the most important ones are missing direct tests for browser cache modules, data table request API coverage, system stats UI coverage, and tool-level LLM system editor specs.
